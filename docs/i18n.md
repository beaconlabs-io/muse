# Internationalization (i18n)

muse uses [`next-intl`](https://next-intl.dev/) for UI translations.
Supported locales: **`en`** (default) and **`ja`**.

## Wiring

- `i18n/routing.ts` — defines `locales = ["en", "ja"]`, `defaultLocale = "en"`,
  and re-exports the typed `Link`, `redirect`, `usePathname`, `useRouter`,
  `getPathname` helpers from `createNavigation(routing)`. Use these helpers
  (not the raw Next.js ones) so locale prefixes are preserved.
- `i18n/request.ts` — resolves the request locale, falls back to the
  default, and loads `messages/{locale}.json` dynamically.
- `messages/en.json`, `messages/ja.json` — translation dictionaries. Keys
  are shared 1:1 between the two files.
- `app/[lang]/…` — all user-facing routes live under a locale segment so
  the locale is part of the URL (e.g. `/en/canvas`, `/ja/evidence/08`).

## Agent output language

UI strings are translated via `next-intl`, but the Mastra agents emit
free-form content (logic-model cards, evidence reasoning, chat replies)
in the **user's input language**, independent of the URL locale. See
[mastra-agents.md → Output Language Handling](./mastra-agents.md#output-language-handling)
for the policy and which structured labels stay in English.

For external paper search, `mastra/agents/query-translation-agent.ts` is
the source of truth for detecting non-English input and translating it
into English keywords before hitting Semantic Scholar.

## Adding a translation key

1. Add the key to both `messages/en.json` and `messages/ja.json`.
2. Use it in a server or client component via `useTranslations("namespace")`
   or `getTranslations` from `next-intl/server`.
3. Keep key names stable — they are the contract across locales.

## Adding a new locale

1. Add the code to `locales` in `i18n/routing.ts`.
2. Create `messages/<locale>.json` with the same keys as `en.json`.
3. Optional: update any agent instructions that hard-code language
   examples (grep `messages/`, `routing.ts`, and agent instructions files).
